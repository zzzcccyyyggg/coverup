# 一、总体进展情况

## 1、课题总体进展情况

### 1.1 课题背景与研究目标

本课题面向当前软件工程、智能软件工程与软件质量保障领域中的一个关键问题：如何面向真实代码仓库，利用大模型自动生成测试用例，在提升测试覆盖率的同时，尽量避免生成过程陷入高成本、低稳定性、低解释性的盲目重试。

与传统依赖随机搜索、路径探索、符号执行或变异引导的自动测试技术相比，大模型驱动的测试生成在自然语言理解、样例迁移和复杂 API 使用方面具有明显潜力，但在真实仓库上也暴露出若干新的系统性困难：

1. 覆盖率结果只能告诉系统“哪些代码没有被执行”，却不能直接解释“为什么没有被执行”；
2. 模型生成出的候选测试经常在编译、运行、断言或环境约束层面失败，若系统缺乏恢复机制，只能进行代价较高的再次生成；
3. Python、Go、Rust 等语言在包管理、类型系统、可见性规则、测试框架、编译检查与运行约束等方面差异显著，单一语言内成立的方法并不一定能自然迁移到其他语言；
4. 仅报告覆盖率提升幅度，往往难以说明系统究竟依赖哪些模块取得效果，也难以说明失败案例为何失败。

基于上述问题，本课题的总体目标，是构建一套**面向多语言代码仓库的覆盖率引导自动测试生成与失败恢复系统**。该系统应当同时具备以下能力：

- 对现有测试集进行覆盖率测量，并自动定位覆盖不足的目标代码区域；
- 将覆盖率缺口组织为可调度的多语言补测任务，而不是简单地将整仓库代码交给模型；
- 在提示构建阶段向模型提供与覆盖目标相关的上下文、控制条件与阻塞解释；
- 在测试失败后引入结构化诊断与恢复机制，而不是将失败统一视为“再调用一次模型”的理由；
- 在 Python、Go、Rust 三类语言之上维持统一控制平面，同时保留语言相关的执行、诊断和修复机制；
- 形成既能支撑论文写作、又能支持项目展示和后续成果申报的软件系统形态。

需要特别说明的是，多语言能力并非课题中的附属特性，而是项目要求中的关键交付方向之一。本课题不以“某一语言上的局部验证”为阶段性终点，而是以“在真实异构代码仓库场景中形成统一可运行系统”为目标。因此，当前项目不仅需要在单一语言上证明技术路线可行，更需要完成 Python、Go、Rust 三类语言环境下的覆盖率采集、任务构造、失败恢复与结果交付。

从当前阶段看，课题已经从“方法设想是否可行”的探索期，进入“系统机制是否闭环、实验论证是否可信、成果是否具备交付形态”的深化期。

### 1.5 系统核心结构与当前已完成内容

从系统建设角度看，本课题当前已经形成了比较完整的多语言技术栈。与传统“单次生成—单次执行”的简单流程不同，系统围绕覆盖率驱动任务组织、结构化错误诊断、工具优先修复、轨迹记录和网页展示构建了完整闭环。具体而言，当前已完成以下八个层面的核心技术工作。

#### （1）覆盖率驱动的动态分段与目标任务组织

系统首先对现有测试集进行覆盖率测量，并将未覆盖或覆盖不足的代码区域组织为后续补测任务。该过程并非简单按文件划分，而是面向语义单元构造 `CodeSegment` 任务对象，用于保存：

- 目标文件路径；
- 目标函数、方法或实现块的起止行号；
- 缺失覆盖的行和分支；
- 已执行行集合；
- 需要保留的上下文片段；
- 与该代码段强相关的 import / use 信息。

在 Python 端，系统调用 SlipCover 获取行级与分支级覆盖率数据，再通过 `segment.py` 中的 AST 分析逻辑，将缺失覆盖行回溯到所属函数、异步函数或类定义；当类体过大时，系统会继续向内收缩到更细粒度的函数/方法级片段，并在必要时补充与目标片段直接相关的 import 语句和上下文代码。

在 Go 端，系统通过 `go test -coverprofile` 获取标准 coverage profile，再由专用解析器将区间级覆盖率映射回源文件行号；随后使用 tree-sitter 识别函数声明、方法接收者、类型定义与关键控制流节点，并结合 `infer_branches()` 估计目标代码附近的分支信息，从而构造适合提示注入和执行验证的代码片段。

在 Rust 端，系统通过 `cargo llvm-cov --json` 获取结构化覆盖率结果，再解析为统一的文件—行号—执行状态表示；之后使用 tree-sitter 与 Rust 专用代码分析工具识别函数、`impl` 块、类型定义与关键控制结构，构造覆盖率缺口对应的目标片段，并自动过滤临时测试文件和无关目标。

这一层的作用，在于把“覆盖率报告”转换为“可进入调度与恢复闭环的补测任务”，是系统区别于简单 prompt 工具的重要起点。

#### （2）基于覆盖率阻塞解释的可达性提示机制

覆盖率缺口只能告诉模型“哪里没有被执行”，却无法解释“为什么没有被执行”。为解决这一问题，系统实现了 `CoverageBlocker` 模块，用于从未覆盖目标反推出控制该目标的上层谓词条件、匹配分支或循环条件，并将其整理为可读的结构化提示。

该模块在 Python 端基于标准库 `ast` 抽取 `if`、`match` 等控制结构，在 Go 与 Rust 端则基于 tree-sitter 分别分析 `if`、`switch`、`match`、`if let` 等语言特有结构。对每个阻塞信息，系统会记录：

- 被阻塞的目标行集合；
- 控制谓词所在行号；
- 谓词文本；
- 谓词类型；
- 变量来源与类型提示；
- 对应的自然语言解释。

随后，系统将这些信息格式化为 prompt 中的 `COVERAGE ANALYSIS` 区块，从而把提示从“请覆盖这些行”提升为“这些行被哪些条件挡住，你需要先构造怎样的状态或输入”。这一机制在复杂行为语义片段上具有重要价值，也是系统将覆盖率结果转化为可操作生成任务的关键桥梁。

#### （3）统一的结构化错误诊断中间表示

在真实仓库上，自动生成测试失败往往发生在多个层次：语法错误、导入错误、可见性错误、类型错误、断言失败、运行时 panic、超时等。若系统只把这些失败作为普通日志文本传给模型，后续的调度、修复和实验分析都会变得非常脆弱。

为此，系统设计并实现了统一的 `DiagnosticIR`。该中间表示以冻结数据结构的方式记录每一次编译、运行、覆盖率检查或修复阶段的结果，核心字段包括：

- 语言类型与处理阶段；
- 工具来源，例如 `pytest`、`go test`、`cargo`；
- 成败状态；
- 归一化错误类别；
- 具体错误码与摘要消息；
- 错误位置；
- 建议修复项；
- 覆盖率变化；
- token 成本、运行耗时、命令行与种子等复现实验信息。

当前 `DiagnosticIR` 已经覆盖导入、类型、可见性、所有权、语法、断言、panic、timeout、依赖、接口、未知等多类错误类别，并对编译、运行、覆盖率、修复、隔离等阶段进行了统一建模。这一设计使系统上层逻辑可以在不关心语言语法差异的前提下，根据失败层执行调度、修复、记忆更新和轨迹记录。

#### （4）有限步、工具优先的修复闭环

系统当前最重要的技术主线，并不是“模型多生成一次”，而是“失败后先做结构化恢复，再决定是否进入下一轮生成”。为此，项目实现了 `RepairOrchestrator`，并在主循环中引入了 bounded tool-first fixpoint 机制。

其基本过程为：

1. 生成候选测试并执行；
2. 若失败，则先将错误归一化为 `DiagnosticIR`；
3. 根据语言和错误类别触发确定性修复算子；
4. 修复后再次编译/运行；
5. 若仍失败，则在有限次数内继续执行修复；
6. 若修复链耗尽，才退回下一轮模型生成。

当前系统已经实现了一批真实可用的修复算子。

Python 侧包括：

- 导入补全；
- 调用形状修正；
- Click 场景下的 `version_option` 参数重排；
- 断言与 oracle 修复；
- 运行时元数据 patch；
- 上下文状态检查相关修复。

Go 侧包括：

- import 增删与 `go mod tidy` 相关修复；
- 未使用变量标记与清理；
- 无效 public accessor 断言移除；
- brittle output oracle 归一化；
- `DebugFlags` 等命令行输出场景的 expectation canonicalization。

Rust 侧包括：

- import 自动补全；
- Rust 2015 crate 的 `extern crate` 自动注入；
- cargo check 结构化建议提取；
- 基于诊断的导入与模块层自动修复。

当前主循环已经显式记录每次外部尝试中的 `repair_passes`、`fixpoint_exhausted` 和 `tool_fixes`，使修复是否真正工作、工作到哪一层，都能被后续实验和报告直接追踪。

#### （5）辅助控制模块：记忆、调度与语义恢复

在统一主循环中，系统还实现了若干辅助控制模块，用于提升重试的方向性与效率。

`ReflectiveMemory` 用于把早先成功或失败的经验按错误类别和上下文签名组织为可检索条目，以便在后续同类错误场景中提供更稳定的恢复建议。  
`UCBPlanner` 用于在多个候选目标片段之间进行焦点分配，减少对明显收益有限片段的盲目投入。  
在 Rust 等更困难语言的持续失败场景中，系统还引入了 semantic recovery mode 等策略，用于在 repeated stall 时把提示从“局部修补”切换为“重新理解实现语义”。

这些模块目前并非项目报告中的唯一主创新，但它们已经实装并真实接入主循环，为后续论文实验中的消融设计和系统成熟度提升提供了基础。

#### （6）执行稳定性、污染防御与工程级约束控制

为避免“生成测试把项目跑坏”或“中间产物污染后续实验”，系统在执行层面增加了多项工程防护措施。

在 Python 端，系统集成 `pytest-isolate`、`pytest-repeat` 与 `pytest-cleanslate.reduce` 等机制，支持隔离执行、重复运行与污染/失败测试的识别处理；在需要时，可自动禁用污染性或持续失败的既有测试，保证覆盖率测量与补测过程更稳定。

在 Go 端，系统对每个 package 使用独立锁，串行化同一包内的临时测试编译与执行，避免并发任务相互覆盖临时文件或制造重复符号错误；同时自动清理残留临时测试文件、统一包名、执行 `gofmt/goimports`，并在必要时用轻量 compaction 去掉多余注释和空行，以满足 backend 的大小约束。

在 Rust 端，系统对 crate 级临时测试执行加锁，自动清理临时测试模块，检测 crate edition，并在必要时执行 edition-aware 导入修复；同时支持 `cargo check --message-format=json` 形式的结构化诊断，以较低代价获取更可操作的编译修复建议。

这一类工程措施虽然不一定单独构成论文主创新，但对真实仓库的持续运行、结果可复现性和网页展示稳定性至关重要。

#### （7）基于运行轨迹的机制证据体系

系统当前不再仅记录“最后覆盖率是多少”，而是对每次外部尝试生成 JSONL 级结构化轨迹。`TraceLogger` 会为每个目标片段记录：

- 尝试编号；
- 当前动作类型，例如 `llm` 或 `tool_repair+llm`；
- 最终结果 `G/F/U`；
- 诊断类别与错误码；
- 覆盖率增量；
- token 和耗时成本；
- 修复轮数与修复耗尽情况；
- 具体执行过的修复算子；
- 是否注入记忆信息。

在此基础上，项目实现了 `trace_matrix.py`、`paper_trace_tables.py` 等脚本，用于把 targeted slice trace 归纳为 paper-friendly 的恢复路径矩阵与 paired baseline 对照表。这意味着当前实验评价已经从“成功案例堆叠”转向“可解释的恢复路径分类”，为论文和项目报告中的机制性结论提供了更可靠的支撑。

#### （8）面向展示与交付的网页平台

除命令行入口外，项目已经完成网页前端与后端控制台，实现仓库上传、参数配置、任务执行、日志查看、结果汇总和测试下载的端到端流程。当前网页平台主要包括：

- 上传源码仓库压缩包；
- 自动检测项目根目录、源码目录与测试目录；
- 选择语言、提示版本、模型与 API key；
- 后台启动补测任务并持续轮询状态；
- 展示初始/最终覆盖率、运行摘要与日志尾部；
- 将本轮新生成且通过验证的测试文件打包下载。

# 二、取得的重要成果

## 1、简要介绍课题研究工作的重要进展、重要成果及应用前景

### 2.1 形成了独立完整的多语言智能测试生成研究系统

本课题当前最重要的成果之一，是已经形成一套独立完整的多语言智能测试生成研究系统。该系统并非简单的提示词包装或界面封装，而是在问题定义、静态/动态分析、失败诊断、修复控制、实验评价与成果交付等方面逐步建立起自洽的研究与工程体系。

对于本课题而言，多语言不是锦上添花的功能补充，而是项目必须完成的重要考核方向。当前系统已经能够在 Python、Go、Rust 三类语言上完成覆盖率测量、目标片段组织、测试执行、失败诊断、恢复控制与结果归档，这意味着项目要求中的多语言支撑能力已经从任务目标转化为阶段性成果。

### 2.2 已在关键创新点和技术点上形成较明确的系统实现

围绕当前主线，项目已经在以下六类核心技术点上取得实质性进展。

#### （1）实现了覆盖率驱动的多语言目标片段构造机制

项目不将整文件或整工程直接送入模型，而是通过覆盖率与语法分析构造语义相对完整、上下文成本可控的目标片段。

Python 端采用 SlipCover + AST 的组合：先获得覆盖率结果，再由 `segment.py` 将缺失行回溯到函数、异步函数或类定义；必要时保留相关 import 和上下文语句，从而生成较紧凑的 `CodeSegment`。  
Go 端采用 `go test -coverprofile` + tree-sitter：先将 coverage profile 映射到行级，再识别函数、方法、receiver 和相关类型定义，并在生成前补齐包信息与目标上下文。  
Rust 端采用 `cargo llvm-cov --json` + tree-sitter：先得到结构化覆盖率，再识别函数、`impl` 块和类型定义，同时结合 crate edition、module API map 等信息形成面向测试生成的上下文。

这一设计使系统能够在控制 token 成本的前提下，尽量保持目标片段的语义完整性，为后续提示构建和恢复分析奠定基础。

#### （2）实现了覆盖率阻塞解释机制，而非仅提供缺失行号

项目进一步实现了 `CoverageBlocker`，用于回答“为什么这些行没有被执行”。这一步不是把未覆盖行号简单打标签，而是尝试上溯其控制条件，抽取阻塞谓词、目标行集合、变量来源和自然语言解释，再将其注入提示。

这一设计的意义在于：系统不再只要求模型“去覆盖某一行”，而是为模型提供关于路径前提、状态约束和分支条件的显式提示，从而把“coverage target”升级为“coverage target + reachability explanation”。

#### （3）构建了统一的多语言结构化诊断中间表示

项目实现的 `DiagnosticIR` 是当前统一控制平面的关键基础。相比把编译错误、运行错误和断言失败全部作为字符串日志处理，`DiagnosticIR` 将其统一为标准化数据对象，使调度、修复、记忆和实验分析能够共享同一套结构。

这项设计带来的直接收益包括：

- 失败类别可以跨语言统一表示；
- 修复算子可以按“语言 × 失败层”注册与调度；
- 运行轨迹可以被离线分析脚本稳定读取；
- 实验不再只回答“最终成功没有”，而能回答“系统在第几层失败、修复是否介入、路径如何变化”。

#### （4）实现了有限步、工具优先的修复闭环

当前系统最具代表性的创新点，是在失败后先执行确定性修复，再决定是否回到下一轮模型生成。围绕这一点，项目实现了 `RepairOrchestrator` 和 bounded tool-first fixpoint。

与传统“失败就重新生成”的做法相比，该机制具有三个直接特点：

1. 某些常见错误可以在低成本条件下先由规则算子吸收；
2. 修复可以在一次外部尝试中连续执行多个 pass，而不是只能做一次修补；
3. 系统能够显式区分“修复未介入”“修复介入但失败”“修复介入并取得正向结果”等不同状态。

这项机制已经在 Python 和 Go 的代表性任务上形成了真实的 repair-assisted positive 事实，并在 Rust 上形成了 prompt-first positive 与 harder contrast 等具有机制意义的结果。

#### （5）形成了基于真实轨迹的实验证据组织方式

项目已经不再满足于“列出若干成功案例”，而是围绕 `TraceLogger`、`trace_matrix.py` 和 `paper_trace_tables.py` 建立了机制证据组织方式。当前系统能够把 targeted slice traces 总结为：

- repair-assisted positive；
- prompt-first positive；
- repair-engaged negative；
- llm-only negative。

这一点对于论文和项目报告都非常重要，因为它意味着项目已经能够区分“哪些收益依赖完整机制”“哪些样例在简化基线下也可成功”“哪些困难样例虽然没有最终正例但已经体现恢复路径变化”。

#### （6）完成了研究系统向展示平台的扩展

项目已经实现网页平台，使系统具备仓库上传、运行配置、任务执行、日志查看、覆盖率展示和有效测试下载等能力。该平台不是静态 UI，而是已经完成真实端到端验证，标志着项目成果已经具备展示与交付价值。

### 2.3 形成了面向多语言场景的分层技术路线

本课题并未把多语言理解为“同一 prompt 换三种语法”，而是把问题重新定义为：**如何在统一控制平面下吸收不同语言的主导失败层**。目前已经形成如下较清晰的技术认识。

#### （1）Python：调用形状、框架 API 与 oracle 问题最突出

Python 端目前成熟度相对最高。系统已经打通了基于 SlipCover 的覆盖率采集、AST 目标片段抽取、pytest 风格测试执行和若干具有代表性的确定性修复。当前 Python 端最具代表性的恢复族包括：

- Click 场景下的参数顺序与调用形状修复；
- 断言与 expected exception 表达修复；
- 运行时元数据 patch 和框架上下文状态相关修复。

从实验现象看，Python 的正向结果更多体现为：先通过工具修复清除可执行性障碍，再由后续生成或再验证把失败推进到更深的断言/语义层，最终转化为正向覆盖率增量。

#### （2）Go：行为前提、输出通道与 CLI 语义问题更突出

Go 端已经完成从“0 到 1”的完整链路打通，并在若干典型 CLI 项目样例上形成了具有研究意义的现象。Go 端当前最具特色的技术点包括：

- 通过 tree-sitter 抽取方法接收者、类型定义和命令行相关上下文；
- 对同一 package 内临时测试执行串行化，避免并发任务互相污染；
- 输出 oracle 归一化与 brittle line-based assertion 修复；
- 针对 public accessor 假设错误、未使用绑定和命令输出格式的确定性修复。

从机制上看，Go 样例清楚显示了完整系统可能出现三类不同结果：repair-assisted positive、prompt-first positive 和 repair-engaged negative。因此，Go 对课题的意义不仅在于“支持了一种语言”，更在于为系统提供了行为语义层面的复杂挑战。

#### （3）Rust：导入、类型、edition 约束与更深层语义恢复仍然最具挑战

Rust 端是当前最困难的语言，也是最能检验系统边界的部分。当前 Rust 端已经完成：

- 基于 `cargo llvm-cov` 的覆盖率采集；
- tree-sitter 级代码分析；
- crate root / crate edition 检测；
- cargo check 结构化诊断解析；
- Rust 2015 crate `extern crate` 注入等导入修复。

从现有 targeted evidence 看，Rust 已经出现至少一个完整系统优于简化基线的代表性事实，但总体成熟度仍低于 Python 和部分 Go 样例。换言之，Rust 当前既是多语言交付能力的一部分，也是系统边界和后续优化方向的最重要来源。

### 2.5 覆盖率已成为正式实验设计中的核心结果指标

在当前论文与实验设计中，覆盖率不是附属指标，而是与机制证据并列的核心结果维度。项目已明确将覆盖率结果分为三个层次组织。

#### （1）最终覆盖率

用于回答完整系统最终是否提升了项目测试覆盖水平。

#### （2）相对原始测试集的覆盖率增量

后续正式实验将统一报告相对于原始仓库测试集的 coverage delta，以避免只报告最终数值而忽略不同项目初始条件差异。

#### （3）覆盖率与恢复路径、成本的联合分析

后续正式实验不会只写“覆盖率更高”，而会同时报告：

- 终态有效/失败/无效数量；
- 修复是否介入；
- 平均修复轮次与修复耗尽情况；
- 成本与单位有效结果代价；
- 不同语言的主导失败层构成。

这种实验设计能够使“覆盖率有没有提高”与“为什么提高或为什么没有提高”在同一框架中得到回答。

### 2.6 已形成正式实验方案，为论文和结题提供支撑

当前项目已经形成一套与论文主张一致的正式实验方案。实验章节将围绕四个核心问题展开：

1. 与简单重试基线相比，完整系统是否改变了恢复路径；
2. bounded tool-first fixpoint 和阻塞解释分别贡献了什么；

## 2、重点阐明课题研究中的重点、难点及突破方法，并说明所产出的专利、论文、软著等知识产权情况

### 3.1 课题研究中的重点

#### 重点一：如何把覆盖率缺口转化为可调度、可恢复、可验证的补测任务

本课题的重点不只是“能不能生成一段测试代码”，而是系统能否围绕真实覆盖率缺口持续推进，并最终形成经过验证的测试结果。因此，覆盖率采集、动态分段、上下文提取与目标片段组织，是整个项目最基础也最关键的技术环节。

#### 重点二：如何在多语言条件下维持统一控制平面

Python、Go、Rust 在包管理、可见性规则、测试约定和错误形式上均存在明显差异。本课题的重点，是在这些差异之上建立统一的任务控制逻辑、统一的错误表示和统一的恢复入口，同时允许语言相关机制保留必要个性。

#### 重点三：如何让失败处理从“日志文本”升级为“可控恢复过程”

课题真正的技术难点不在于模型能否输出测试代码，而在于系统能否对失败做出结构化反应。因此，`DiagnosticIR`、`RepairOrchestrator`、bounded fixpoint 和 trace 体系，是项目区别于简单提示工程的关键。

#### 重点四：如何让实验结果真正支撑方法主张

本课题的实验重点，不是堆叠若干成功案例，而是建立能够解释系统何时有效、为何有效、哪些收益依赖完整系统、哪些并不依赖完整系统的证据链。

#### 重点五：如何把研究系统扩展为可展示、可交付成果

除研究逻辑外，项目还需要面向汇报、答辩、演示和成果转化完成平台化封装。因此，网页平台、结果打包、真实端到端运行链路等工程工作也构成当前阶段的重要工作内容。

### 3.2 课题研究中的难点

#### 难点一：覆盖率反馈与语义单元之间并非天然对齐

覆盖率报告给出的是文件和行号，而测试生成需要的是函数、方法、`impl` 块、控制条件与上下文片段。如何将“行号级反馈”稳定映射到“语义片段级任务”，并在多语言场景下维持统一抽象，是项目的基础难点。

#### 难点二：失败具有层次性、连续性和语言差异性

测试生成失败并不是单一事件，而是语法、导入、可见性、类型、断言和语义问题逐层暴露的过程；不同语言上最显著的失败层也并不一致。系统若不能识别这一点，就会陷入代价高昂且不可解释的盲目重试。

#### 难点三：确定性修复本身也可能引入新的错误

项目当前一个很重要的认识是：修复算子不是天然正确的。修复若实现不严谨，可能把原有失败推进到新的假错误层，甚至污染后续运行。因此，修复正确性本身就是系统设计与实验评价的一部分。

#### 难点四：多语言工程实现中存在大量执行稳定性问题

例如 Python 测试污染、Go 包内并发临时文件冲突、Rust crate 级临时模块注册与 edition 差异、网页平台与后台任务隔离等，均不是单纯 prompt 调整能够解决的问题，而需要工程级约束与运行治理。

#### 难点五：论文叙事与工程实现必须同步收敛

随着系统规模扩大，工程上容易出现模块众多、机制复杂的情况。项目报告与论文撰写必须避免“模块堆积式叙事”，而应围绕少量稳定核心主张组织材料，这是当前课题的重要总结难点。

### 3.3 本课题的主要突破方法

#### 突破方法一：建立覆盖率到语义片段的多语言映射链路

课题通过 Python AST、Go/Rust tree-sitter、coverage parser 与分支推断逻辑，形成了从覆盖率结果到 `CodeSegment` 的统一映射链路，使动态覆盖率反馈真正能够驱动多语言补测任务。

#### 突破方法二：引入覆盖率阻塞解释与结构化诊断

课题将“未覆盖”与“失败”这两个原本分离的问题连接起来：前者通过 `CoverageBlocker` 解释为什么尚未触达目标，后者通过 `DiagnosticIR` 统一表示失败层级，从而使系统具备面向路径前提与失败层的双重解释能力。

#### 突破方法三：引入有限步、工具优先的恢复机制

课题通过 `RepairOrchestrator` 和 bounded fixpoint，使系统能够在再一次调用模型之前，优先尝试低成本、可验证的规则修复。这一机制显著提升了系统的可解释性，也让多语言恢复逻辑第一次具备了统一的控制入口。

#### 突破方法四：建立基于运行轨迹的机制证据体系

系统不再只记录最终结果，而是记录完整恢复路径、修复介入、修复轮次、修复耗尽情况和失败层变化，并通过 `trace_matrix.py` 与 `paper_trace_tables.py` 等工具对其进行标准化整理，使实验具备机制解释能力。

#### 突破方法五：通过网页平台打通研究、展示与交付

网页平台的建设使项目从内部研究工具转向更可演示、可交付的软件系统，为中期汇报、成果答辩、软著申报和后续转化提供了现实基础。

### 3.4 典型实验结论与当前边界

当前项目已经获得若干可以正式用于汇报的阶段性结论：

1. 完整系统已经在 Python 与 Go 的代表性任务上形成了修复介入后取得正向结果的事实；
2. 部分 Go 样例即使在简化基线下也能成功，说明项目没有将所有收益都归因于完整机制；
3. 在更困难的 Go 行为语义样例上，完整系统虽未全部形成最终正例，但已将恢复路径从简单失败推进为结构化介入；
4. Rust 作为当前最复杂语言，已出现至少一个完整系统优于简化基线的代表性对照事实；
5. 网页平台已通过真实端到端验证，表明项目具备展示和交付属性。

同时，也应明确当前边界：

- 尚不能宣称已经形成稳定的跨语言全面领先结果；
- 尚不能宣称所有语言上的修复机制成熟度完全一致；
- 尚不能将当前 targeted 实验等同于最终投稿级大规模实验。

保持这种边界意识，并不削弱项目价值，反而是当前阶段结论可信的重要基础。
